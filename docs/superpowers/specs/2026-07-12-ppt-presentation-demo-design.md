# 競賽簡報呈現與系統展示影片 — 設計文件

> 2026-07-12。決賽簡報（8-10 分鐘）的敘事結構、每模組 demo 影片分鏡、與 Playwright 自動化錄製架構。
> 產出兩類：腳本文件（`docs/presentation/簡報腳本.md`）+ 影片成品（`demo-videos/*.mp4`，9 支）。

---

## 1. 背景與目標

- 競賽簡報不做現場 demo，系統展示以**影片嵌入 PPT** 呈現；簡報時影片無聲播放、講者現場講解。
- 每個功能的呈現節奏固定：**為什麼需要（why）→ 我們怎麼做（how）→ 系統展示（demo 影片）**。
- 其餘頁面後端仍在開發中，錄製採「**現在全部錄一版、後端上線後重錄對應段**」策略——因此錄製必須是**可重複的自動化腳本**（腳本即真相，重錄免費）。
- 講稿中的參考數字與名詞以**報告書 v6**（`內文V6/報告書_完整版_v6_加入引用連結.pdf`）為準，撰寫腳本文件時取用。

## 2. 決策紀錄

| 問題 | 決定 | 理由 |
|---|---|---|
| 簡報時間 | 8-10 分鐘 | 競賽規格 |
| 本輪產出 | 腳本文件 + 影片實際錄製 | 影片由自動化錄出，可重複 |
| 敘事主軸 | 按模組逐一講（why→how→demo），最後用數位員工以故事收官串起生態系 | 結構清楚好準備；收官呼應 agent「工具鏈操作六模組」的設計定位 |
| 六模組時間分配 | 均分（各 ~60s），把各自特色展現出來 | 使用者定案；節奏一致換取評審預期感，變化留給收官段 |
| 影片標註 | 純畫面不加字 | 講者現場講解，畫面乾淨 |
| 錄製策略 | 現在全部錄一版（live 用真後端、mock 照現況），後端上線後單支重錄 | 先有完整一版可排練 PPT，重錄成本低 |
| Agent 收官態 | 真 Gemini live（標題列 GEMINI LIVE 入鏡） | 說服力最強；不確定性以多 take 對策 |
| 錄製技術 | 方案 A：Playwright 錄製腳本（進版控、合成游標、headed 真 GPU） | 與「上線後重錄」策略相性最佳；方案 B（MCP 驅動+螢幕錄影）不可重複、方案 C（人工錄）重錄靠人，皆捨 |

## 3. 簡報整體結構與時間軸（目標 9 分 45 秒，上限 10 分）

| # | 段落 | 時長 | 內容 | 影片素材 |
|---|---|---|---|---|
| 0 | 開場 hook | ~60s | 三大時代命題（IMO 淨零談判重啟、EU ETS 2026 全面履約、航港局航運產業升級方案 24.5 億）→ 痛點一句話：「港口的資料是孤島，決策靠人工」 | `hero-cover.mp4`（~10s loop：波浪影片+標題+七 chips） |
| 1 | 系統架構 | ~45s | 一頁講清生態系：「六大功能 + 一位數位員工」、資料交換層、live/mock 雙態設計 | `hero-overview.mp4`（~10s：Enter→總覽儀表牆轉場） |
| 2-7 | 六模組逐一講 | 6×60s | 每模組固定節奏：why 15s → how 15s → demo 影片 30s；順序照 rail 編號：碳權→政策→孿生→派工→疫情→警報 | 六支 ~30s clip |
| 8 | 數位員工收官 | ~90s | 故事式：前面六個模組變成一位 AI 員工手上的工具，一句話指令跨模組完成任務 | `agent-finale.mp4`（~75s，GEMINI LIVE） |
| 9 | 結尾 | ~30s | 量化效益回收 + iMarine 資料使用對齊 + 願景一句話 | 無（靜態頁） |

合計 585s ≈ 9 分 45 秒。排練若超時，優先壓縮開場與結尾（六模組節奏與收官故事不動）；開場三命題各一句話帶過，細節留 Q&A。

**貫穿主軸**：全簡報需有一句主軸句，開場提出、結尾回收（例：「把高雄港的資料孤島，變成一個會自己思考的生態系」，實際文字於腳本文件定稿）。評審連看多隊，一句可複述的主軸是記憶點（調研沉澱，見 §11）。

設計理由：

- 六模組順序照 rail 編號 1-6——影片中 UI 左側 rail 高亮順序與簡報一致，警報（6）講完接數位員工（7）也是系統內的畫面順序，敘事與介面互相印證。
- 固定 60 秒節奏是刻意的：均分之下用節奏一致換取評審的預期感，把變化留給收官段的 90 秒故事，形成對比。
- hero 頁直接當 PPT 開場底圖——它本來就是為 PPT 開場設計的。

## 4. 六模組 demo 分鏡（每支 ~30 秒）

每支影片統一節奏：**0-5s 進頁定場**（評審看清版面與資料源 chip）→ **5-25s 二至三個互動** → **最後 5s 停在 payoff 畫面**（講者收結論時畫面停在最有說服力的一格）。

| 模組 | why（講者痛點） | demo 分鏡（錄什麼） | payoff 停格 |
|---|---|---|---|
| 1 碳權（live） | 減碳成效無法變現、EU ETS 履約成本壓力 | 進頁（LIVE chip 入鏡）→ 發行一筆 SU（真上鏈）→ 訂單簿掛單→成交 | 鏈上確認 + 餘額變動 |
| 2 政策 | IMO 談判反覆，幕僚彙整政策情報耗時數天 | 點 chip 生成報告（生成動畫 + 情報流入）→ 綜合對話問一題 | 帶引用來源的回答 |
| 3 孿生（live） | 泊位調度靠老師傅經驗，看不見未來 24 小時 | 港區 2.5D 全景（船隻動態）→ 拖 24hr 時間軸推演 → 泊位佔用變化 | 未來時刻港區狀態 |
| 4 派工 | 天氣窗口決定作業安全，法規/慣例散落人工判讀 | 風險大字塊+一句話結論 → 切到颱風情境（全版轉玫紅）→ 燈號矩陣變紅 → 點列展開法規依據 | 派工卡「停什麼、加派什麼」 |
| 5 疫情 | 疫情從海上來，人工比對航跡×疫區不可能即時 | 船隊清單 → 下鑽高風險船 → 地圖航線+疫區熱點 → 拖時間游標（命中脈衝）→ 按「模擬偵測」 | 新威脅 85 分紅級自動置頂 |
| 6 警報 | 從偵測到民眾手機的最後一哩延遲 | 按模擬事件（颱風紅色警報）→ 港區 cell 全亮+波紋 → 手機全螢幕插播 → 送達漏斗滾數字 | 手機上的紅色警報插播 |

原則：

- 每支只選「這個模組獨有」的互動（派工的法規展開、疫情的時間游標、警報的兩段式演練），對應「均分但各展特色」的定案。
- 警報放六模組最後，視覺最強，講完銜接數位員工：「偵測、預警、推播都自動了——有沒有人把這一切看住？」
- **評分項對齊（撰寫講稿時強制執行）**：每模組 how 的 15 秒需點名該模組使用的 iMarine 資料集與 AI 技術——評審第一件事是核對評分項（iMarine 使用比例 10% + AI 輔助 5% 是明碼），散在六模組講比結尾一頁彙總更有說服力；結尾頁再放一張評分項對照收攏。
- 分鏡設計優先讓互動落在**視覺大元素**（KPI 大字、modal、手機 mock、燈號矩陣），投影幕距離下小字看不清；關鍵小字時刻（如訂單簿數字）可用至多一處**靜態 zoom cut**（ffmpeg crop 放大切鏡，非動態 pan），此為版面手法非標註，不牴觸「純畫面不加字」定案。
- 誠實註記：carbon 與 twin 的細部操作路徑（內部分頁名稱、按鈕位置）以**實際畫面為準**，撰寫錄影腳本時逐一探明微調；本表為意圖層級分鏡，不預先鎖死 selector。

## 5. 數位員工收官分鏡（~75 秒，GEMINI LIVE）

故事主軸：**「前面六個模組是六套專業工具；數位員工把它們變成一位 7×24 的值班員——人只在關鍵決策點按確認。」**

| 時間 | 畫面 | 講者主旨 |
|---|---|---|
| 0-10s | 進頁 → 開場自動巡檢：7 張燈卡逐一亮起 → 招呼泡泡帶健檢結論 + 3 建議 chips | 上工第一件事：自我巡檢，六個模組健康狀態即時回報 |
| 10-18s | 輸入綜合指令：「幫我盤點六大模組狀態整理成今日戰情摘要，另外看碳權市場現況，把我持有的 SU 挑一批掛單上架」 | 一句話，跨模組的任務 |
| 18-32s | plan-then-act：3 步計畫渲染 → 工具卡逐一執行，右欄工作區跟隨顯示真實模組數據卡 | 它自己拆解計畫、呼叫各模組的資料層 |
| 32-42s | 回答帶模組色 citation chips → 點一顆跳到對應模組頁 → 返回，對話完整保留 | 每個結論都可溯源到模組現場 |
| 42-62s | 互動掛單卡：SU 下拉挑真實持有的代幣 → 改總價、折合每噸即時換算 → 按確認 → 真上鏈（held→listed） | 寫入操作一定經過人的確認——human-in-the-loop |
| 62-75s | 完成回覆 + SUGGEST 追問 chips 浮現 → 停格（GEMINI LIVE chip 入鏡） | 它還會建議你下一步；這就是生態系合在一起的樣子 |

- 指令刻意涵蓋讀（多模組盤點）+ 寫（掛單）兩類工具，讓巡檢、工具卡、citation、確認卡、SUGGEST 五個 UX 自然全出現。
- 掛單接 carbon 真後端、真上鏈，複證第一段碳權模組的 live 能力，首尾呼應。
- Live 不確定性：同一腳本跑 2-3 個 take 挑最好的；等待回應的空檔是誠實的，過長等待以 ffmpeg 修剪。指令文字可由講者在錄製時改為台上要唸的版本。

## 6. 錄製技術架構

```
scripts/demo/
  recorder.ts        共用 runner：起獨立 dev server（專用埠）→ 開 headed Chromium
                     （真 GPU，twin WebGL 與 Mapbox 品質所需）→ 注入合成游標
                     （平滑移動 + 點擊漣漪）→ 1920×1080 錄影 → webm 轉 mp4（ffmpeg）
  scenarios/         每支影片一個腳本，共 9 支：
                     hero-cover / hero-overview / carbon / policy / twin /
                     dispatch / epidemic / alert / agent-finale
```

- `playwright` 為 **devDependency**（不進 bundle，`npm run build` 產物零影響）。
- 合成游標：headed 錄影不含 OS 游標，注入 overlay（圓點 + 點擊漣漪、easing 移動軌跡）呈現操作感。**節奏定量規則**（調研沉澱：緩慢、有目的、先懸停再點擊）：移動 600-900ms easing、懸停 300-500ms 再點擊、點擊漣漪 ~400ms，具體數值實作時以 probe 影片目檢微調。
- 後製（ffmpeg，輕量）：修剪頭尾死幀；載入/等待段**速度 ramp 1.5-2x**（正常互動不加速）；必要時的靜態 zoom cut（見 §4）。不做轉場特效、不加字。
- 單支重錄一鍵：`npm run demo:record -- <scenario>`；後端上線後只重錄該段。
- 錄製隔離（沿用 CDP 驗收慣例）：獨立 dev server 埠 + 獨立 browser profile，**不動使用者的 `.env`、不碰既有 :5173/:8000**；agent live 的 Gemini key 以環境變數傳入錄製用 dev server，不入鏡、不進版控。
- 影片規格：mp4 / H.264 / 無音軌（Keynote 與 PowerPoint 皆原生支援）。解析度預設 1920×1080；**實作第一步先錄 10 秒 probe 驗證文字銳利度**，不合格則提高錄製解析度（如 DSF 2 產出 4K 下轉 1080p，兼取 zoom cut 餘裕）或改桌面擷取，擇一定案後全部 scenario 沿用。

## 8. 前置條件與風險

| 風險 | 對策 |
|---|---|
| carbon 與 agent 掛單需 PoC 後端（:8000/:8545），且錄製會真上鏈、污染鏈上數字 | 每次錄製前 `make chain` 重起 = 乾淨狀態，數字可重現 |
| Gemini live 每次回答不同、可能不照預期呼叫工具 | agent-finale 跑 2-3 take 挑最好；過長等待 ffmpeg 修剪 |
| policy 的 rag-agent（:8100）未上線 | 先錄 mock fallback 現況版，上線後重錄（策略已定） |
| epidemic/alert 需 Mapbox token 與連網 | `.env` 既有 token；錄製環境需連網 |
| ffmpeg 本機是否可用 | 實作第一步檢查，缺則 `brew install ffmpeg` |
| mock 頁面標題列顯示 MOCK chip 入鏡 | 照實入鏡（系統設計即有資料源 chip）；後端上線重錄後自然轉 LIVE |
| Playwright webm 錄影畫質可能偏軟 | 10 秒 probe 先驗（見 §6），不合格改高解析度或桌面擷取，全部 scenario 統一方案 |
| 簡報現場影片播放失敗 | 備援雙保險：影片檔另存隨身碟；每支影片的 payoff 停格輸出成截圖放 PPT 附錄頁，影片掛掉可用截圖講完（調研沉澱：never without a backup plan） |

## 9. 驗收標準

1. `docs/presentation/簡報腳本.md` 涵蓋段落 0-9 全部：每段講稿要點、秒數、對應影片檔名、影片分鏡（步驟級）、**cue 表**（講到哪句按播放）、**每模組 iMarine 資料集與 AI 技術對照**（供 how 段講稿與結尾對照頁取用）。
2. 9 支 mp4 全部產出，逐支人工目檢通過（畫面完整、游標軌跡自然、payoff 停格正確、無 console 錯誤畫面入鏡），最終由使用者看一輪。
3. 每支影片的 payoff 停格另輸出一張截圖（PPT 附錄備援用，與影片同名 `.png` 存 `demo-videos/stills/`）。
4. 三綠燈不受影響：`tsc --noEmit` 0、`vitest run` 全綠、`build` 成功（錄影腳本不進 `src/`）。
5. 重錄可行性驗證：任選一支**確定性 scenario**（agent-finale 以外，該支為 Gemini live 本質不確定）連跑兩次，產出影片內容節奏一致。

## 10. Out of scope

- PPT 頁面本身的製作（版面、slide 設計）——本輪只出腳本與影片素材。
- 影片旁白、字幕、標註字卡（定案：純畫面）。
- 報告書內容修改。
- 各模組後端功能開發（policy rag-agent、dispatch/epidemic/alert live provider）。

## 11. 調研沉澱（外部參考）

設計中標注「調研沉澱」處的依據，2026-07-12 網路調研：

- **評審視角**（Devpost 評審訪談）：評審第一件事是核對評分項要求；storytelling 是投入度關鍵；「完成且可用的產品感」勝過單點驚豔；影片開頭幾秒就要讓評審知道這是什麼（評審連看多隊）。→ 落入 §3 主軸句、§4 評分項對齊原則、每支影片 0-5s 定場。
- **Demo 影片工法**（screen-recording 專業指南）：游標要慢、有目的、先懸停再點擊，加 click highlight；載入等待段加速 1.5-2x、正常互動不加速；剪掉死幀與冗餘點擊；桌面觀看為主時提高錄製解析度以留 zoom 餘裕；產品 demo 甜蜜點 60-90 秒。→ 落入 §6 游標節奏定量規則、後製規範、解析度 probe；§4 zoom cut 後備。
- **Pitch 現場**：影片播放必須有備援方案（never without a backup plan）。→ 落入 §8 備援雙保險、§9.3 停格截圖。
- 來源：Devpost「6 Tips for making a winning hackathon demo video」「How to win a hackathon: Advice from 5 seasoned judges」、ngram「Turn Screen Recordings Into Professional Demos」、Alejandro Cremades「How To Embed A Video In A Pitch Deck」等（連結見對話紀錄，內容已沉澱於本節，不依賴外鏈存活）。
